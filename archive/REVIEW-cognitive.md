# 认知过程验证：管道通了，但脑子还没接上

## 上一轮验证的盲区

REQUIREMENTS-CHECK.md 验证的是：

```
"Agent 能不能用 git show 读到 v2 的参数？" → 能
"Agent 能不能用 git diff 对比两个版本？"   → 能
"Agent 能不能用 ctx_update 写入？"        → 能
```

这只是验证了**管道**（基础设施能跑通）。

但认知记忆系统的核心不是管道，而是**过程**——
信息如何被捕获、何时被整理、如何被检索、怎样被遗忘。
这些过程在当前设计中**大部分处于欠设计状态**。

---

## 逐一检验认知过程

### 1. 即时编码 — "什么时候记？记什么？"

**设计现状**：
- 提供 ctx_update 工具
- system prompt 说"遇到重要信息时立即记录"

**问题**：
"重要信息"的定义是什么？对谁来说重要？

同一轮对话中：
```
用户: "用 MA 10/30 跑一下，哦对了最大回撤不能超过 25%，
       然后记得用 2020-2023 的数据。"
```

这里有三条信息：
- MA 参数 (strategy-params) ← 明显重要
- 回撤约束 (constraints) ← 容易被忽略但很关键
- 数据范围 (backtest config) ← 可能被遗忘

Agent 会记住哪些？全凭它的判断。
System prompt 说"遇到重要信息时记录"，但**没有定义"重要"的标准**。

**缺失的设计**：编码触发规则

```
应该被编码的信息类型（写入 system prompt）：
1. 用户明确表达的约束/要求 → constraints.md
2. 参数/配置的变更 → 对应领域文件
3. 测试/实验结果 → 对应结果文件
4. 决策及其理由 → decisions.md
5. 用户明确说"不要"/"必须" → constraints.md
6. 发现的 bug 或异常行为 → 领域文件

不需要编码的：
- 中间推理过程（LLM 的思考）
- 临时的代码尝试（可以重新生成）
- 用户的闲聊
```

没有这个规则，编码过程是随机的。

### 2. 记忆固化 — "整理的 LLM prompt 够用吗？"

**设计现状**：
- §5.2 写了 "LLM 调用: 基于新对话，更新 Context 文件"
- 列了几条 prompt 指令（增量更新、焦点切换标记、大小控制）

**问题**：
这是整个系统**最关键的环节**，但设计中只用了几行 bullet points 描述。

实际需要回答：

**a) LLM 的输入如何组织？**

```
输入 = 当前 context 文件 + 序列化的对话

但如果有 5 个 context 文件 × 平均 1k tokens = 5k tokens
加上对话 60k tokens
= 65k tokens 的输入

LLM 需要在这 65k tokens 中：
- 识别对话中的新信息
- 判断它属于哪个文件
- 决定是新增、修改还是不变
- 输出结构化的更新
```

这是一个非常高难度的任务。prompt 需要非常精细。

**b) LLM 的输出如何可靠地解析？**

§5.3 用了 XML 格式，但没讨论：
- 如果 XML 格式不对怎么办？
- 如果 LLM 在 content 中也用了 `<file>` 标签怎么办？
- 如果 LLM 遗漏了 `_index.md` 的更新怎么办？

**c) 信息衰减问题**

每次 compaction 是一次"传话游戏"：
```
compaction 1: 对话 → Context v1 (信息可能丢失 10%)
compaction 2: Context v1 + 新对话 → Context v2 (再丢失 10%)
compaction 3: Context v2 + 新对话 → Context v3 (再丢失 10%)
...
compaction 10: 累积丢失可能达到 65%
```

每次 LLM 更新 context 文件，都可能"精炼"掉一些细节。
10 次 compaction 后，早期的信息可能已经面目全非。

**git 历史能缓解但不能完全解决**——因为 Agent 通常读的是
当前版本的 context 文件，不会每次都去 git history 核查。

**缺失的设计**：对抗信息衰减的机制

可能的方案：
- 固化字段：某些信息一旦写入就标记为 `[pinned]`，不允许精炼/删减
- 周期性校验：每 N 次 compaction，让 LLM 对比 current context vs 早期版本，检查是否有信息丢失
- 可量化的 context 健康度指标

### 3. 注意力管理 — "焦点切换的判断谁来做？"

**设计现状**：
- Agent 调用 ctx_update 时带上 Active/Paused 状态
- compaction prompt 中说"焦点切换时标记状态"

**问题**：
焦点切换的判断完全靠 LLM。但焦点切换并不总是显式的：

```
显式: "先放下均线策略，研究动量策略" → 容易识别
隐式: 用户开始问一个新话题，没有说"放下旧话题" → 不容易识别
渐变: 讨论逐渐从话题 A 过渡到话题 B → 什么时候算切换了？
```

**没有严重问题**，但 system prompt 需要指导 LLM 如何判断：

```
焦点切换判断规则：
- 用户明确说"先做X"/"放下Y" → 显式切换
- 用户连续 3 轮讨论新话题，且未提及旧话题 → 可能切换，询问用户
- 用户在同一轮中讨论多个话题 → 不切换，所有话题保持当前状态
```

### 4. 自动回忆 — "最重要的功能，设计最少"

**设计现状**：
- §6.2 有 6 行伪代码
- "关键词提取 + Context 历史匹配"
- Phase 3 实现

**问题**：
我们在 COGNITIVE-FRAMEWORK.md 中说这是**最重要的功能**
（因为 Agent 的元认知不可靠），但设计中它的篇幅最少。

需要回答的关键问题：

**a) 触发条件是什么？**
每次用户输入都触发？还是只在检测到"可能需要历史信息"时触发？

每次触发：成本高（每轮都搜索 + 可能注入），但不会遗漏。
选择性触发：成本低，但如何判断"可能需要历史信息"本身就是一个难题。

**b) 搜索范围是什么？**
搜索当前 context 文件的 git 历史？
搜索 commit messages？
搜索 context 文件的内容？

**c) 注入多少？**
找到 3 个相关 commit，每个 context 文件 2k tokens，
注入 6k tokens → 吃掉大量 context 空间。

需要一个注入预算（如 max 2k tokens）和优先级排序。

**d) 不注入比乱注入好**
如果注入了不相关的历史信息，Agent 可能被误导。
精确率比召回率更重要。

### 5. 遗忘/归档 — "什么时候从 Hot 变成 Warm？"

**设计现状**：
- §4.3 描述了 Hot/Warm/Cold 三层
- "超限时启动分层精简"
- "长期 Paused 精简为一行引用 [see commit hash]"

**问题**：
"长期"是多久？"超限"的阈值是什么？谁执行这个判断？

```
是 LLM 在 compaction 时判断？
  → 需要在 prompt 中给出明确规则（如"超过 2 次 compaction 未被引用"）
  → LLM 可能不计数，不可靠

是确定性代码判断？
  → 需要追踪每个 topic 的最后访问时间
  → 在 compaction hook 中，代码先计算出哪些 topic 应该归档
  → 然后在 LLM prompt 中明确指示"请将以下 topic 精简为引用: ..."
```

**确定性 + LLM 混合**更可靠：
代码做判断（哪些该归档），LLM 做执行（怎么写精简后的内容）。

### 6. 粒度切换 — "zoom 的过程是否顺畅？"

**设计现状**：
- Level 3 (_index.md) → Level 2 (文件) → Level 1 (git 历史) → Level 0 (JSONL)

**问题**：
Level 0 (回到 JSONL) 的路径没有实现。

如果 Agent 需要"用户当时的原话"，context 文件里只有精炼后的信息。
回到 JSONL 需要知道对应的 entry ID，但 context 文件中没有记录 entry ID。

**需要决定**：
- 是否在 context 文件中保留 JSONL entry ID 作为溯源？
  如 `[ref: entry-a1b2c3d4]`
- 还是认为 Level 0 极少需要，不支持？

## 总结：管道 ✅，认知过程 ⚠️

| 认知过程 | 基础设施 | 过程设计 | 状态 |
|---------|---------|---------|------|
| 即时编码 | ctx_update 工具 ✅ | 编码触发规则 ❌ 缺失 | ⚠️ |
| 记忆固化 | compaction hook ✅ | 固化 prompt 详细设计 ❌ 不足 | ⚠️ |
| 信息衰减对抗 | git 历史 ✅ | 防衰减机制 ❌ 缺失 | ⚠️ |
| 注意力管理 | 状态标注 ✅ | 切换判断规则 ❌ 不足 | 🟡 |
| 自动回忆 | before_agent_start hook ✅ | 触发/搜索/注入策略 ❌ 缺失 | ⚠️ |
| 遗忘/归档 | cold/warm/hot 概念 ✅ | 转换规则和执行机制 ❌ 缺失 | ⚠️ |
| 粒度切换 | 多层级结构 ✅ | Level 0 溯源路径 ❌ 缺失 | 🟡 |

**基础设施全部到位，但认知过程的"规则"和"判断逻辑"大部分缺失。**

这些不是代码问题——它们是**prompt engineering + 确定性规则**的设计问题。
在实现之前需要把这些规则明确下来，否则系统的行为将是不可预测的。
