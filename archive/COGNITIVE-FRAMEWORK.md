# 认知框架：git-mem 本质上是一个 LLM 的认知记忆系统

## 核心认识

人脑和 LLM Agent 面临的是**完全相同的问题**：

> 用有限的工作记忆（~7 个单元 / ~200k tokens），
> 操作远超其容量的知识，
> 在多个任务间切换焦点，
> 同时维护一个持续演化的世界模型。

人脑用了数亿年进化出一套解决方案。
我们在设计 git-mem 的过程中，独立地收敛到了**几乎相同的架构**。
这不是巧合——这说明架构方向是对的。

## 认知科学中的关键机制，以及对设计的启示

### 1. 工作记忆的容量限制与组块 (Chunking)

**认知科学**：Miller (1956) 的经典发现——工作记忆容量约 7±2 个"组块"。
但组块本身可以很大。专家通过把信息组织成有意义的组块来突破容量限制。

象棋大师能记住棋盘布局，不是因为记忆力更强，
而是因为他们把棋子组织成了有意义的模式（组块），
一个模式就是一个组块单位。

**对设计的启示**：Context 文件就是"组块"。

```
没有组块 (原始对话):
  "用户说 MA=10" "用户说 slow=30" "用户说 RSI=70" "回测显示 Sharpe=1.5"
  → 4 个独立记忆项，占据工作记忆

有组块 (Context 文件):
  strategy-params.md: "v2: MA 10/30, RSI 70 → Sharpe 1.5"
  → 1 个组块，包含同样的信息，但工作记忆负担降低
```

**设计原则**：每个 Context 文件应该是一个**有意义的认知组块**——
一组在语义上密切相关的信息，作为一个单元被理解和操作。

不要让文件太碎（每个事实一个文件 → 组块太多），
也不要太大（一个文件包含所有信息 → 回到了没有组块的状态）。

### 2. 编码特异性与情境依赖记忆

**认知科学**：Tulving (1973) 的编码特异性原理——
回忆的效果取决于编码时和检索时的情境匹配。
你在图书馆学习的东西，在图书馆考试时更容易想起来。

**对设计的启示**：**在信息产生的当下捕获，远比事后回忆提取更可靠。**

这正是为什么 MemGPT 式的**实时记录** (`ctx_note`) 
比 compaction 时的**批量提取**更可靠——
编码时的情境（完整的对话上下文）
比检索时的情境（compaction 时部分上下文已经模糊）更丰富。

```
实时记录 (编码情境完整):
  用户说 "最大回撤不超过 25%"
  → Agent 当场有完整的对话上下文
  → ctx_note("constraints", "MaxDD ≤ 25%") ← 高保真编码

批量提取 (编码情境退化):
  20 轮对话后，compaction 时
  → Agent 需要从长对话中回忆这个约束
  → 可能漏掉，因为它不是对话的重点 ← 编码退化
```

**设计原则**：
实时记录是第一道防线（高保真），compaction 校准是第二道防线（查漏补缺）。

### 3. 注意力的聚光灯模型与缩放

**认知科学**：Eriksen & Eriksen (1974) 的注意力聚光灯模型——
注意力像一个可以调节的聚光灯，可以聚焦（窄而深）或扩散（宽而浅）。

你在读代码时，注意力窄而深（逐行理解逻辑）。
你在做架构设计时，注意力宽而浅（整体结构）。

**对设计的启示**：Agent 需要在不同粒度间切换。

```
缩放层级:

Level 3 (最广):  _index.md
                 "项目有 3 个目标，2 个活跃，1 个暂停"

Level 2 (主题):  goals.md / strategy-params.md
                 "均线策略: v2 参数, Sharpe 1.5"

Level 1 (版本):  mem_recall(hash, file)
                 "v2 的完整参数和回测数据"

Level 0 (原始):  mem_recall → JSONL entry (如果需要)
                 "用户当时的原话和完整工具输出"
```

**设计原则**：工具应该让 Agent 能在这些层级间自然地上下导航。
`_index.md` 是全局视图，`mem_recall` 是深入细节。
Agent 根据当前任务需要选择合适的粒度。

### 4. 前瞻记忆与预期性编码

**认知科学**：人脑不只被动记录发生的事，还会**主动预期**即将需要的信息，
并提前从长期记忆中检索。

你走进厨房时，大脑已经在激活"厨房相关"的记忆，
你不需要刻意去"搜索"锅在哪里。

**对设计的启示**：这就是 REVIEW-v2 中提到的**自动回忆**。

```
用户输入: "用 v2 的参数重新跑一下"
    ↓
before_agent_start hook:
    关键词 "v2" + "参数" → 自动检索 Context 历史
    找到 strategy-params.md 的 v2 版本
    → 自动注入到 context 中
    ↓
Agent 不需要主动调 mem_recall，相关信息已经在 context 里了
```

这是**最高优先级**的功能——比 Agent 主动搜索记忆更可靠，
因为它不依赖 Agent 的"元认知"（知道自己不知道什么）。

### 5. 遗忘曲线与间隔效应

**认知科学**：Ebbinghaus (1885) 的遗忘曲线——
记忆随时间指数衰减，但每次回忆都会重置曲线（间隔重复效应）。

**对设计的启示**：我们已有的冷热分层天然模拟了遗忘曲线。

```
Active (刚编码/频繁访问) → 完整保留在 Context 文件中
Paused  (一段时间未访问)  → 精简为引用 + commit hash
Archived (长期未访问)     → 仅在 git 历史中，需要搜索

每次 mem_recall 某个 topic → 该 topic 的"激活"重置
                           → 可以考虑将其提升回 Paused 甚至 Active
```

## 这个认知框架给 git-mem 的终极定位

git-mem 不是"一个存储工具"或"一个 compaction 增强"。

**git-mem 是一个 LLM Agent 的外部认知系统——
它外化了 Agent 原本隐式的心智模型，
使其可持久化、可版本控制、可检索、可回溯。**

| 人脑的隐式过程 | git-mem 的显式实现 |
|---------------|-------------------|
| 你"记住"了某件事 | `ctx_note` 写入 Context 文件 |
| 你"想起"了某件事 | `mem_recall` 或自动回忆注入 |
| 你"忘记"了某件事（但它还在潜意识里）| 归档到 git 历史（可恢复） |
| 你"改变了看法" | Context 文件更新 + git diff 记录变化 |
| 你"切换注意力" | `ctx_status` Active → Paused |
| 你"同时记住多件事" | 多个 topic 共存，不同状态 |
| 你"反思总结"一天的经历 | compaction 校准 |
| 你"回想过去的版本" | `mem_recall(old_hash)` |
| 你"对比两种方案" | `mem_diff(hash_a, hash_b)` |

人脑做这些事时是隐式的、不完美的（会真的遗忘、会扭曲记忆）。
git-mem 让这些过程变得显式的、可靠的（git 不会遗忘、不会扭曲）。
